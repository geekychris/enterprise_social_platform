# WorkSphere Scalability Architecture

## Current State

### Messaging

The messaging system uses a PostgreSQL-backed conversation model:

```
conversations (id, name, type, created_by)
    вҶ“ 1:N
conversation_participants (conversation_id, user_id, last_read_at, visible_from)
    вҶ“ 1:N
messages (id, conversation_id, sender_id, content, read, created_at)
```

**How messages are delivered today:**
- Clients poll `GET /conversations` every 10 seconds for the conversation list
- When viewing a thread, clients poll `GET /conversations/{id}/messages` every 3 seconds
- No push, no WebSocket, no SSE for message delivery
- Unread counts are computed by scanning all messages where `created_at > last_read_at`

**How messages are sent:**
1. Client POSTs message вҶ’ saved to PostgreSQL вҶ’ response returned
2. Bot check runs asynchronously on a virtual thread
3. Other participants discover the message on their next poll cycle (0-3 second delay)

**N+1 query problem:** Listing conversations requires 1 base query + 2 queries per conversation (latest message + unread count). With 50 conversations: ~101 queries.

### Feed

The feed is assembled on every request:

```
FeedService.assembleFeed(userId, cursor, limit)
  1. Query all followed user IDs + own ID
  2. Query all group/page membership IDs
  3. SELECT posts WHERE authorId IN (...) OR (targetType IN (...) AND targetId IN (...))
  4. Filter by visibility in-memory
  5. Get recommendations from RecommendationService (trending + FOF + cross-team)
  6. Score: engagement Г— recency_decay Г— affinity_boost
  7. Interleave: 1 recommended per 5 organic posts
  8. Apply cursor-based pagination
```

**Recommendation scoring:**
```
score = (reactions Г— 1.0 + comments Г— 2.0) Г— 0.5^(hours/24) Г— (1.0 + affinity Г— 0.3)
```

Sources: trending posts (7-day window), friend-of-friend content (via AOEE graph), cross-team popular content.

### Database

Single PostgreSQL instance. Posts/comments are partitioned by quarter. Messages, conversations, reactions are not partitioned. IDs generated by an in-memory atomic counter per object type (not distributed).

### Current Scaling Limits

| Component | Limit | Bottleneck |
|-----------|-------|------------|
| Message delivery | ~100 concurrent users | Polling every 3s = 33 req/s per conversation viewer |
| Conversation list | ~50 conversations/user | N+1 queries (101 queries for 50 conversations) |
| Unread count | O(total_messages) | Full scan across all conversations |
| Feed assembly | ~500ms per request | Multiple DB queries + in-memory sort + recommendation engine |
| Recommendations | ~1s per request | AOEE graph calls + DB queries + scoring |
| ID generation | Single process only | In-memory AtomicLong, resets on restart |
| Bot responses | ~15s latency | Ollama LLM call is synchronous/blocking |

**WebSocket Gateway** (dedicated service):
- Maintains persistent connections per user (one connection per device)
- Authenticates via JWT on connect
- Subscribes to Redis channels: `user:{userId}:messages`, `conversation:{convId}:typing`
- When a message is saved, the Message Service publishes to Redis вҶ’ Gateway pushes to all connected devices

**Connection management:**
- Each gateway instance holds ~50K connections (Netty/Vert.x, not Spring MVC)
- Horizontally scalable: add more gateway instances behind a load balancer
- Sticky sessions via consistent hashing on userId (so reconnects go to same gateway)
- Heartbeat every 30s to detect stale connections

**Message flow:**
```
1. Client sends message via WebSocket
2. Gateway forwards to Message Service (REST or gRPC)
3. Message Service writes to PostgreSQL
4. Message Service publishes to Redis: PUBLISH conversation:{convId} {messageJson}
5. All Gateway instances subscribed to that conversation's channel
6. Gateway pushes message to all connected participants
7. Delivery latency: <100ms
```

**Fallback for disconnected devices:**
- On reconnect, client sends `last_seen_message_id`
- Gateway queries Message Service for messages since that ID
- Returns catch-up batch (max 100 messages)
- Uses the `conversation_participants.last_read_at` as a fallback cursor

#### 1.2 Message Storage Optimization

**Partitioned messages table:**
```sql
ALTER TABLE messages PARTITION BY RANGE (created_at);
-- Monthly partitions (more granular than posts)
-- Messages are write-heavy, read-recent-only
```

**Denormalized conversation summary:**
```sql
CREATE TABLE conversation_summaries (
    conversation_id  BIGINT PRIMARY KEY,
    last_message_id  BIGINT,
    last_message_at  TIMESTAMPTZ,
    last_message_preview TEXT,  -- First 100 chars
    last_sender_id   BIGINT,
    updated_at       TIMESTAMPTZ
);
-- Updated via trigger or application on every message insert
-- Eliminates the per-conversation "latest message" query
```

**Unread count materialization:**
```sql
CREATE TABLE unread_counts (
    user_id          BIGINT,
    conversation_id  BIGINT,
    unread_count     INT DEFAULT 0,
    PRIMARY KEY (user_id, conversation_id)
);
-- Incremented on message insert (if sender != user)
-- Reset to 0 on mark-read
-- Total unread = SUM(unread_count) WHERE user_id = ?
-- O(number_of_conversations), not O(total_messages)
```

**Cache-aside pattern:**
1. Check Redis first
2. If miss: query PostgreSQL, write to Redis with TTL
3. On write: invalidate Redis key (or update if possible)

**User profile cache saves ~90% of DB reads** вҖ” profiles are fetched on every message DTO conversion.

#### 1.4 Offline/Reconnect Protocol

For mobile devices that frequently go offline:

```
Client reconnects вҶ’ sends: {
  last_conversation_list_version: "v123",
  conversations: {
    "conv_1": { last_message_id: 12345 },
    "conv_2": { last_message_id: 67890 }
  }
}

Server responds вҶ’ delta: {
  new_conversations: [...],
  updated_conversations: [
    { id: "conv_1", new_messages: [...], unread: 3 },
    { id: "conv_2", new_messages: [...], unread: 1 }
  ],
  removed_conversations: [...]
}
```

**Key design principles:**
- Client tracks last-seen state per conversation
- Server computes delta, not full state
- Batched response: one request catches up everything
- Maximum catch-up window: 7 days (older messages require explicit load)
- Compressed response (gzip) to minimize bandwidth

### Phase 2: Distributed Feed Architecture

#### 2.1 Pre-Computed Feed (Fan-Out on Write)

Instead of assembling the feed on every read, pre-compute it:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ     в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ     в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  Post        в”Ӯв”Җв”Җв”Җв”Җв–әв”Ӯ  Feed Fan-Out в”Ӯв”Җв”Җв”Җв”Җв–әв”Ӯ  Redis Sorted в”Ӯ
в”Ӯ  Service     в”Ӯ     в”Ӯ  Worker       в”Ӯ     в”Ӯ  Sets (per    в”Ӯ
в”Ӯ              в”Ӯ     в”Ӯ  (Kafka)      в”Ӯ     в”Ӯ  user feed)   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ     в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ     в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**When a post is created:**
1. Post saved to PostgreSQL
2. Event published to Kafka topic `posts.created`
3. Fan-out worker consumes event:
   - Gets all followers of the author
   - Gets all members of the target group/page
   - For each eligible user: `ZADD feed:{userId} {score} {postId}`
4. Score = timestamp (for chronological) or computed relevance score

**When a user loads their feed:**
1. `ZREVRANGE feed:{userId} 0 19` вҖ” top 20 posts by score
2. Look up post details from PostgreSQL (or Redis cache)
3. No expensive assembly, no visibility filtering (already done at fan-out time)

**Fan-out trade-offs:**
- **Small-audience posts** (personal, small groups): fan-out to all followers (fast)
- **Large-audience posts** (company announcements, 10K+ followers): "fan-out on read" вҖ” store in a special "high-reach" feed and merge at read time
- **Hybrid approach** (like Twitter): fan-out for users with <5K followers, merge on read for celebrities/company accounts

**Signals for feed ranking:**

| Signal | Source | Weight | Description |
|--------|--------|--------|-------------|
| Engagement | reactions + comments | High | More engagement = more interesting |
| Recency | created_at | High | Exponential decay, 24h half-life |
| Author affinity | reaction history | Medium | How often user engages with this author |
| Content affinity | embedding similarity | Medium | Semantic match with user's interests |
| Social proximity | graph distance | Medium | 1st degree > 2nd degree > 3rd degree |
| Group relevance | membership activity | Medium | Active groups ranked higher |

**Two-stage ranking:**
1. **Candidate generation** (fast, 1000 candidates): Pull from pre-computed feed + trending + recommendations
2. **Scoring** (precise, top 100): Apply full feature extraction and ML model
3. **Re-ranking** (policy, top 20): Apply diversity rules, dedup, content policy

#### 2.3 Embedding-Based Content Matching

Use the existing Ollama `nomic-embed-text` model for semantic matching:

```sql
-- Add pgvector extension
CREATE EXTENSION vector;

-- Add embedding column to posts
ALTER TABLE posts ADD COLUMN embedding vector(768);

-- Index for similarity search
CREATE INDEX idx_posts_embedding ON posts USING ivfflat (embedding vector_cosine_ops);
```

**Embedding pipeline:**
1. On post creation: generate embedding via Ollama `nomic-embed-text`
2. Store in PostgreSQL pgvector column
3. At feed time: compute user interest vector (average of their recent reactions' post embeddings)
4. Find semantically similar posts: `ORDER BY embedding <=> user_vector LIMIT 50`

#### 3.2 Database Sharding Strategy

**User-based sharding for messaging:**
```
Shard key: conversation_id % num_shards

Shard 0: conversations 0, 4, 8, ...    вҶ’ PostgreSQL instance 0
Shard 1: conversations 1, 5, 9, ...    вҶ’ PostgreSQL instance 1
Shard 2: conversations 2, 6, 10, ...   вҶ’ PostgreSQL instance 2
Shard 3: conversations 3, 7, 11, ...   вҶ’ PostgreSQL instance 3
```

All messages for a conversation live on the same shard вҶ’ no cross-shard queries for thread reads.

**User's conversation list** requires a cross-shard query вҶ’ use a separate index table:
```sql
-- On each shard:
CREATE TABLE user_conversations (
    user_id          BIGINT,
    conversation_id  BIGINT,
    shard_id         SMALLINT,
    last_activity_at TIMESTAMPTZ,
    PRIMARY KEY (user_id, conversation_id)
);
```

**Content sharding:**
- Posts already partitioned by time вҶ’ extend to time + author_id sharding
- Or: keep content on a single large PostgreSQL cluster with read replicas
- Recommendation data in a separate read-optimized store (Elasticsearch/OpenSearch)

#### 3.3 Distributed ID Generation

Replace the in-memory `GlobalIdGenerator` with a distributed scheme:

**Snowflake-style IDs:**
```
64-bit ID:
  [1 bit sign] [41 bits timestamp] [10 bits node_id] [12 bits sequence]

- Timestamp: milliseconds since epoch (69 year range)
- Node ID: unique per service instance (1024 instances)
- Sequence: per-millisecond counter (4096 IDs/ms/node)
```

**Benefits:**
- No central coordinator
- Roughly time-ordered (helps with cursor pagination)
- Each service instance generates IDs independently
- Compatible with existing Int64 columns

## Migration Path

### Step 1 (Weeks 1-4): Foundation
- Add Redis caching (user profiles, conversation lists, unread counts)
- Denormalize conversation summaries (eliminate N+1)
- Materialize unread counts
- **Impact: 5-10x reduction in DB load**

### Step 2 (Weeks 5-8): Real-Time Messaging
- Deploy WebSocket gateway (Netty-based)
- Add Redis Pub/Sub for message delivery
- Implement offline catch-up protocol
- **Impact: <100ms message delivery, eliminate 33 req/s per active user**

### Step 3 (Weeks 9-16): Feed Pipeline
- Deploy Kafka
- Implement fan-out on write for feed
- Move feed storage to Redis sorted sets
- Build two-stage ranking pipeline
- **Impact: Feed loads in <50ms from cache, ranking quality improves**

### Step 4 (Weeks 17-24): Service Split
- Extract Messaging Service (with its own DB)
- Extract Feed Service
- Deploy API Gateway
- Implement distributed ID generation
- **Impact: Independent scaling, 10x capacity per service**

### Step 5 (Ongoing): ML and Intelligence
- Add pgvector for semantic search
- Train feed ranking model on engagement data
- Implement personalized notification scheduling
- A/B testing framework for feed experiments
- **Impact: 2-3x engagement improvement from better ranking**
